# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
make build              # build binary (or: go build -o rosterbot .)
make install            # install to $GOPATH/bin
make test               # run all unit tests (or: go test ./internal/...)
go test ./internal/optimizer/...  # run a specific package's tests
make dry-run            # run optimizer locally without applying changes
go run . optimize --dry-run --dates 2026-04-01  # test a specific date
go run . optimize --dry-run --dates 2026-03-26:2026-03-28  # test a date range
go run . optimize --dry-run --dates all  # test full season from today
go run . optimize --dry-run --matchup    # test remaining days in current matchup period
go run . prospects --dry-run  # run prospect report locally
go run . gs-check --dry-run          # check league-wide GS violations for the most recently completed period
go run . transactions --dry-run      # check recent trades with HKB valuations
go run . waivers --dry-run           # Statcast-driven free-agent picks (top 15 by default)
go run . waivers --dry-run --top 25 --positions OF,SP   # bigger list, position filter
go run . backtest                                    # backtest last completed matchup week
go run . backtest --dates 2026-04-13:2026-04-19      # backtest a specific window
go run . backtest --skip-projections                 # lineup-only backtest (faster)
go run . optimize --dry-run --snapshot               # write projection snapshots in dry-run (non-dry-run runs write by default)
go run . recap --out /tmp/recap.html                 # render weekly HTML recap (most recent completed week)
go run . recap --dates 2026-04-20:2026-04-26 --out /tmp/recap.html  # specific window
go run . recap --out /tmp/recap.html --open          # render and auto-open in default browser
go run . recap-site --out dist                       # render every completed week into a static site dir
go run . recap-site --out dist --open                # build site and auto-open dist/index.html
go run . shadow                                      # capture today's projections from all 4 RoS systems (read-only; for model comparison)
go run . shadow --dates 2026-06-30                   # capture a specific date
go run . archive --dry-run           # capture today's ephemeral data (HKB, projections, Savant, prospects); --dry-run prints sizes
go run . archive --date 2026-06-30   # capture a specific date
make clean-cache        # rm -rf .cache/  (cold-pass baseline before make run-all)
make run-all            # exercise every command in dry-run / read-only mode + print cache size
```

**`make run-all` is the canonical end-to-end smoke test** — it iterates every CLI command in dry-run / read-only mode with `time` on each step and prints the final `.cache/` size. Use it for two things: (1) a single-command sanity check before pushing changes, and (2) observing cache behavior — stderr `cache hit:` / `cache miss:` lines show what each command touched. **Whenever you add a new top-level CLI command (a new `cmd/<x>.go` registered on `rootCmd`), append a corresponding line to the `run-all` recipe in the `Makefile`** so the smoke target stays comprehensive. Pair with `make clean-cache && make run-all` for a cold pass, then `make run-all` again to see warm-cache behavior.

After making code changes, always run `go vet ./...` and `go mod tidy` to catch issues early. Note: `gofmt` and `go vet` run automatically via PostToolUse hooks on every Edit/Write.

Tests require no credentials — all network dependencies are mocked via interfaces or test servers.

For local dev, create a `.env` file (gitignored) with `FANTRAX_USERNAME`, `FANTRAX_PASSWORD`, `FANTRAX_LEAGUE_ID`, `FANTRAX_TEAM_ID`, `FANTRAX_IL_SLOTS`, `FANTRAX_MINORS_SLOTS`. Loaded automatically by `godotenv`.

Optional env vars with defaults: `GS_TRACKING_ENABLED` (false) — enables games-started tracking, used by both the optimizer (weekly GS budget) and gs-check (league-wide violation detection). The real min/max are always fetched live from Fantrax's own per-period `getTeamRosterInfo?view=GAMES_PER_POS` configuration (see `internal/fantrax.GetGSLimits`) — there's no numeric env var for the limit itself, since Fantrax scales it whenever a period spans more than one calendar week (season opener, All-Star break). `PROSPECT_ROLLING_DAYS` (14), `PROSPECT_MIN_GAMES` (8), `PROSPECT_RANK_CACHE_HOURS` (168), `PROSPECT_UPGRADE_RANK_THRESHOLD` (20).

GS-check env vars (required only for `gs-check` command): `GS_TRACKING_ENABLED=true`, `PUSHOVER_GROUP_KEY`, `PUSHOVER_API_TOKEN`. If the live GS-limit fetch fails, both `optimize` and `gs-check` send a Pushover alert via `PUSHOVER_USER_KEY`/`PUSHOVER_API_TOKEN` (the personal ops-alert channel, same as the cache's stale-fallback notification) rather than falling back to a stale static number.

## Architecture

The optimizer runs as a single binary (`main.go`) with Cobra subcommands (`cmd/`) that wire together four independent packages:

```
fantrax client  ──┐
mlb schedule    ──┼──► optimizer ──► apply lineup (or dry-run print)
fangraphs proj  ──┘
```

**`internal/cache`** — generic TTL cache using Go generics (`FileCache[T]`). Stores JSON with a `fetched_at` timestamp envelope behind a byte-level **`Store`** seam. `Get(key, fetchFunc)` returns cached data if fresh, otherwise calls `fetchFunc`, saves, and returns. All I/O errors are non-fatal. TTL of 0 bypasses cache (`--no-cache` flag). Storage is pluggable: `fsStore` (default; `<dir>/<key>.json` flat under `.cache/`) and `MemStore` (tests) live in the zero-dep leaf; the S3 adapter `s3store` (`internal/cachestore/s3store`, aws-sdk-go-v2) lives in its own package so the AWS SDK stays out of the leaf. `cache.SetDefaultStore(store)` swaps storage process-wide for every `FileCache` without touching the ~30 `New[T](dir, ttl)` call sites (mirrors the `Verbose`/`Notify` package globals). On AWS, `cmd` calls `SetDefaultStore(s3store…)` keyed under the `cache/` prefix when `STATE_BUCKET` is set, so the bot reads/writes the cache per-key live to S3 and `entrypoint.sh` no longer syncs `cache/`. See `docs/adr/0001-s3-not-db-for-cache.md`.

**Caching at a glance**: three TTL tiers map to three mutability classes.

| Tier | TTL | Use for | Examples |
|------|-----|---------|----------|
| `pastPeriodTTL` | 30 d | Past-period snapshots (immutable once the period closes) | `fantrax-roster-stats-<teamID>-<period>`, `fantrax-pitcher-gs-<teamID>-<period>`, `fantrax-recent-stats-pitcher-<teamID>-<period>` (hitter recency now comes from the `fantrax-roster-stats` daily series, not a dedicated snapshot), `fantrax-{hitter,pitcher}-roster-<teamID>-<period>` (when period < current), `mlb-schedule-<YYYY-MM-DD>` (past dates), `mlb-player-id-<name>-<team>` |
| `stableTTL` | 7 d | Season-invariant config | `fantrax-{hitter,pitcher}-slots-<leagueID>`, `fantrax-{hitter,pitcher}-scoring-<leagueID>`, `fantrax-season-range-<leagueID>` |
| `todayTTL` | 15 m | "Today, but stable for a window" — drifts during the day, but fine to reuse for an hourly GHA loop or local-dev iteration | `fantrax-{hitter,pitcher}-roster-<teamID>`, `fantrax-current-period-<leagueID>-<YYYY-MM-DD>`, `fantrax-player-pool-<leagueID>`, `fantrax-minors-roster-<teamID>`, `fantrax-available-prospects-<leagueID>`, `fantrax-pending-trades-<leagueID>`, `fantrax-all-trades-<leagueID>`, `fantrax-all-transactions-<leagueID>` |

A handful of provider-specific TTLs sit outside this scheme: `fangraphs-{bat,pit}-*` (24 h, `projections.ProjectionCacheTTL`), `mlb-handedness` (7 d), `savant-*` (24 h, `statcast.CacheTTL`), `hkb-*` (8 h), `prospect-rankings-*` (168 h via `PROSPECT_RANK_CACHE_HOURS`), `mlb-game-logs-<playerID>-<group>-<season>` (1 h, prospects MiLB game log), `mlb-game-log-<playerID>-<group>-<season>` (1 h, MLB-only game log used by `BackfillDailyFPts`). These predate or sit between the three-tier scheme and use whatever TTL fits the upstream's update cadence. The FanGraphs and Savant TTLs are **single exported constants** (`projections.ProjectionCacheTTL` / `statcast.CacheTTL`, both 24 h matching the once-daily upstream cadence) so every consumer (optimize, backtest, waivers, claims) shares one freshness policy on the shared cache entry — TTL is a per-read max-age against the stored `fetched_at`, not baked into the key, so divergent TTLs across callers would otherwise cause needless refetches.

**Cache key naming**: `<source>-<entity>[-scope...]` where `source` ∈ {`fantrax`, `mlb`, `fangraphs`, `savant`, `hkb`, `prospect`, `mlb-handedness`}, `entity` is the data shape (e.g. `roster-stats`, `pitcher-gs`, `matchups`, `schedule`, `bat`, `pit`), and `scope` is whatever disambiguators apply (teamID, period, leagueID, date). The repeated prefixes are named constants owned by the package that builds them — the whole `fantrax-*` / `mlb-game-log` family in `internal/fantrax/cachekeys.go`, plus `keySavant` (`statcast/load.go`) and `keyFanGraphs` (`projections/fangraphs.go`) — so a prefix rename lands in one place; genuine single-use keys (`mlb-schedule`, `mlb-handedness`, `mlb-player-id(s)`, `mlb-game-logs`) stay inline literals. Old `backtest-snapshot-mlb-*` files (pre–roster-stats rename), `backtest-snapshot-*` files (StatsType=2 era), and `player-ids.json` (pre-FileCache prospects bulk file) are orphaned and safe to delete.

**Claims cursor and ledger**: `.cache/last-claims.json` is a small JSON file written by `internal/claims` after each successful run; it records the most recent transaction timestamp seen so the next run fetches only new events (cursor-based incremental delivery). The audit ledger is written to `.waivers/claims/<date>.json` (one file per UTC calendar date); the `.waivers/claims/` directory is created on first write and is safe to include in the GHA cache alongside `.cache`.

**`Client.SetCache(cacheDir)`** in `internal/fantrax/client.go` enables the file cache for every cached method on `*fantrax.Client`. `cmd/root.go`'s `initApp` calls it on every command's client unless `--no-cache` is set, so commands inherit caching automatically — no per-command opt-in. When `cacheDir` is empty, every cached helper short-circuits to its uncached `fetchXxx` sibling, which is also how tests stay hermetic. Per-period methods (`GetHitterRosterForPeriod`, `GetRecentPitcherStats`, etc.) call `c.ttlForPeriod(period)` — past periods get `pastPeriodTTL`, current/future fall back to `todayTTL`. The "current" bound comes from `PeriodForDate(seasonStart, time.Now().UTC())` against the cached `fantrax-season-range`, so there's no extra upstream call to determine the cutoff.

The matchups response is also memoized in-memory per `*fantrax.Client` (via `allMatchups()`) — five different helpers consume the same `GetAllMatchups` payload during a single `recap-site` build, and in-memory is the right scope since the in-progress week mutates intra-day.

**`internal/config`** — loads env vars via `godotenv`, validates that all four required vars are set, and returns a `Config` struct used by the CLI commands to wire everything together.

**`internal/scoring`** — the single home for the stat→fantasy-points algebra. `ApplyHitter`/`ApplyPitcher` take a neutral `HitterLine`/`PitcherLine` of raw counts and apply `Weights` (the `fantrax.ScoringWeights` alias). Pure, imports nothing else in the tree, so `fantrax` (game-log backfill) and `projections` both depend on it without a cycle. See the **Scoring model** section below for the adapter pattern.

**`internal/positions`** — the single source of truth for Fantrax position-ID semantics: ID constants (filling `"003"`/`"008"`, which `auth_client` omits), `SlotName`, `AcceptsINF`, `IsPitcherSlot`, and `HitterBucket` (C>INF>OF>UT reporting precedence). A leaf package (depends only on `auth_client`); `fantrax` and `backtest` import it instead of hardcoding position-ID switches.

**`internal/fantrax`** — wraps `github.com/pmurley/go-fantrax` (public read API) and `go-fantrax/auth_client` (authenticated API + lineup writes). Key details:
- `auth_client` uses chromedp (headless Chrome) to log in and obtain a session cookie. Cookie is cached in `.fantrax-cache/`. On first run or cache miss, a browser opens.
- Credentials read from env: `FANTRAX_USERNAME`, `FANTRAX_PASSWORD`, `FANTRAX_LEAGUE_ID`, `FANTRAX_TEAM_ID`.
- **Fantrax API version** — every `/fxpa/req` POST body must include `"v": "<current>"`. The library centralizes this in `fantraxAPIVersion` in `auth_client/fantrax_client.go`. When Fantrax deploys a new version and calls start returning `STALE_CLIENT` (empty `responses` array), probe the API with `curl` to find the new version string — update the constant in the library and also in `gs_check.go`, which builds a `/fxpa/req` payload directly. (`daily_fpts.go` now goes through `auth_client`'s request builder and carries no version of its own.)
- **Position IDs are numeric strings** (`"001"` = C, `"002"` = 1B, `"003"` = 2B, `"004"` = 3B, `"005"` = SS, `"008"` = INF, `"012"` = OF, `"014"` = UT). These come from the roster API and must be used as-is for slot assignment and eligibility checks. Their semantics live in **`internal/positions`** (the single source of truth): ID constants — filling `"003"`/`"008"`, which upstream `auth_client` omits — plus `SlotName`, `AcceptsINF`, `IsPitcherSlot`, and `HitterBucket` (the C>INF>OF>UT reporting precedence). `fantrax` and `backtest` both import it instead of hardcoding switches.
- This league's active slot names: `C`, `1B`, `2B`, `3B`, `SS`, `INF`, `OF` (×4), `UT` (×3). Mapped in `posNameToID` in `client.go` (which references `positions` constants).
- Scoring group code is `BASEBALL_HITTING` (not `HITTING`).
- **Scoring periods are daily** (period 1 = season opener), though Fantrax merges consecutive days into one period number around multi-day breaks (e.g. the All-Star break) and can insert extra daily periods mid-season (doubleheader/postponed-game makeups) — so plain `1 + days since season start` arithmetic can drift. Matchup data from `GetAllMatchups()` has weekly matchup entries, not daily — don't use it for period lookup.
- **Two distinct period axes, two distinct helpers — do not cross them.** The two axes are distinct Go named types — `WeeklyPeriod` and `DailyPeriod` (`period_types.go`) — so passing a weekly value where a daily one is required (or vice versa) is a compile error, not a silent bug (rosterbot-1i3, the fix for the rosterbot-uv6 / rosterbot-z3b bug class). `GetScoringPeriodsAndTeams()` returns `[]ScoringPeriod` parsed from Fantrax's standings `SCHEDULE` view captions ("Scoring Period N (date - date)"); despite the name, `N` is the **weekly matchup** number (~7 days/period, merged wider around breaks — e.g. period 16 spans 2026-07-13..07-26 across the All-Star break, per `TestFindCurrentPeriod_MergedAllStarBreakPeriod`). `FindCurrentPeriod(periods, date)` / `FindJustEndedPeriod` answer "which weekly period contains/just-ended for this date" from that list — correct for GS-limit lookups (`GetGSLimits` takes the weekly number) and standings-style questions. Roster/apply/GS-snapshot endpoints (`GetHitterRosterForPeriod`, `GetPitcherRosterForPeriod`, `ApplyLineup`, `getPlayerGSSnapshotForPeriod`) are keyed by a completely different, **daily** period number (advances one per calendar day, e.g. 104…110 across a week) that Fantrax never exposes as a list — only as "the current period" via `GetCurrentPeriod()`. **`fantrax.DailyPeriodFor(currentPeriod, seasonStart, today, date)`** (`gs_period_walk.go`) is the single source for that axis: anchors on `GetCurrentPeriod()`'s value via `AnchorPeriodForDate` when known (exact within an insertion-free window around today), else falls back to `PeriodForDate(seasonStart, date)` day math. `GetTeamGS`/`GetTeamPitcherStarts` walk each day of a (weekly) `ScoringPeriod` via **`gsPeriodWalk`**, which calls `DailyPeriodFor` per day — never the weekly `ScoringPeriod.Number` itself, which would make every day in the period fetch the identical snapshot. `internal/lineuprun.Run`'s per-date apply loop (`optimize`/`shadow`) resolves each `--dates`/`--matchup` date to its daily period the same way, via `DailyPeriodFor` directly — **not** via `FindCurrentPeriod`/the weekly `periods` list, and not via a `periods`-first resolver.
  - A prior change (**rosterbot-uv6**, commit 92ed478) mistakenly routed `GetTeamGS`'s daily walk through a periods-list-first resolver (`fantrax.ResolvePeriod`, since deleted); every day then fetched the *same* weekly-numbered snapshot, collapsing all day-over-day deltas to zero and under-counting each team's weekly GS to ~one day's worth. That surfaced as a whole-league false "Under Min" GS alert at the first All-Star-break boundary (2026-07-13) and was reverted to daily numbering (**rosterbot-wd5**).
  - The exact same class of bug independently recurred in the lineup-apply path: `rosterbot-z3b` (2026-07-12) introduced `fantrax.ResolvePeriod` — a periods-list-first, three-tier resolver intended as "the single entry point for mapping a calendar date to its scoring period number" — and wired `internal/lineuprun.go`'s per-date apply loop through it. Its tier 1 trusted the weekly `periods` list as if it were daily-grained; since virtually every in-season date falls inside some weekly period's date range, tier 1 won on nearly every call. When the All-Star break merged two weeks into period 16 (2026-07-13..07-26), every `--matchup` date in that 11-day span resolved to the identical period 16: `GetHitterRosterForPeriod`/`GetPitcherRosterForPeriod` fetched the same stale snapshot for all 11 dates, `ApplyLineup` wrote all 11 dates' changes to the same period, and — because the diff for each date's "actual" target day never converged — the same lineup swap re-applied and fired a fresh Pushover notification for all 11 dates on **every hourly `Lineup` run**, the flood of duplicate lineup-change alerts diagnosed 2026-07-14. Fixed by switching the apply loop to `DailyPeriodFor` directly and deleting `ResolvePeriod` (no safe production use case remained: every real caller needs the daily axis, and the weekly `periods` list can't provide it). The baseline-day lookup (`dayBeforePeriod`) resolves via the same `DailyPeriodFor` daily numbering. Historical/backtest-facing period lookups (`DailyFantasyPoints`, `GetTeamPitcherStarts`, Class B) are **resolved** (**rosterbot-ren**) via the authoritative season-wide date→daily-period map: `Client.dailyPeriodForDate` consults `periodDateMap` (`period_date_map.go`), parsed once from `getTeamRosterInfo`'s `DisplayedLists["periodList"]` dropdown (`"104 (Mon Jul 6)"` … — the label for period N *is* the date whose snapshot lives at period N, so it's self-correcting across Fantrax's mid-season period insertions), cached season-stable (`fantrax-period-date-map-<leagueID>-<year>`, `stableTTL`) plus in-memory memoized like `allMatchups`; it soft-falls-back to naive `PeriodForDate` on any miss/fetch error, so hermetic tests and credential-less renders are unaffected. The Class-A anchored path (`DailyPeriodFor`/`AnchorPeriodForDate` for lineup-apply + `GetTeamGS`) is **unified onto the same authoritative map** (**rosterbot-2ax**, closed 2026-07-16): `DailyPeriodFor` is now a `*Client` method that consults `c.periodDateMap` first and only falls back to `AnchorPeriodForDate`/`PeriodForDate` on a map miss (date outside coverage) or nil client (hermetic tests pass `nil`, matching the pre-fix anchor/day-math behavior exactly). The "identical output today — zero drift" assumption the deferral was filed under broke the very next day: on 2026-07-16, live `GetCurrentPeriod()` returned 113 for that date while the freshly-refetched (not stale-cached) periodList map said 113 = 2026-07-15 and 114 = 2026-07-16 — Fantrax's own "current period" pointer lagged a full calendar day behind its periodList dropdown. The lineup-apply hot path submitted changes against the already-closed period 113; Fantrax rejected the affected players as "already locked," and `applyLineupWithLockedPlayerRetry`'s locked-player retry masked the resulting no-op as "Lineup applied successfully" (a separate follow-up: that retry path trusts an empty/benign post-retry response instead of verifying the intended change actually landed — filed as **rosterbot-48z**). `gsPeriodWalk` gained a leading `*Client` param (nil-safe) to thread the map through to `GetTeamGS`/`GetTeamPitcherStarts`.
- **Future lineup apply** requires a two-step confirmation flow: first API call returns a confirmation prompt (`ShowConfirmWindow=true`), second call with the same payload applies the changes. Handled in `ApplyLineup`.

**`internal/projections`** — FanGraphs projections via the `--projections` flag. Supported systems: `steamer`, `depthcharts`, `thebatx`, `atc`, plus their rest-of-season variants `steamer-ros`, `depthcharts-ros`, `thebatx-ros`, `atc-ros`. Default is `depthcharts`; the bot currently runs on `depthcharts-ros` in season. Wraps the configured system with a rolling-stats fallback chained via `ChainedSource`. FanGraphs returns **JSON** (not CSV); player name field is `PlayerName`. The `Projection` struct includes derived stats (`Singles`, `XBH`, `TB`) that must be computed from raw fields before scoring. Separate `Source` (hitters) and `PitcherSource` (pitchers) interfaces.

**Blended scoring** — wraps the configured FanGraphs source with recent Fantrax stats. The recent signal differs by role. **Hitters** use a **trailing 30-day window**: `internal/lineuprun`'s `windowedHitterRecent` reconstructs the per-day FPts series (`DailyFantasyPoints` over the last ~35 days, backfilled) and collapses it via `projections.WindowedRecent` + `WindowWeight(30)` into per-player FP/game + games-in-window, as of today. **Pitchers** still use **season-to-date** YTD FP/G from a single roster snapshot: the Fantrax roster API returns cumulative YTD `FantasyPointsPerGame` regardless of the period requested, so `GetRecentPitcherStats` reads the latest completed period's snapshot. Both fall back to 100% base projection when no recent data. Weights are **dynamic** (not fixed) and shift toward recent stats as a player accumulates a stable sample. The hitter window **replaced** unbounded YTD after `backtest --recency-experiment` (**rosterbot-2nd**) showed w30 beating unbounded YTD by ~1 realized pt/game/day over both the full season (109d) and the last 8 weeks — raw YTD double-counts the in-season signal already regressed into `depthcharts-ros`. Pitchers showed **identical** lineup outcomes across every strategy (pitcher slotting is game-availability-bound, not projection-bound), so they stay on YTD. The harness compares base / ytd / 14d / 30d / decay variants for **both** roles; see `docs/superpowers/specs/2026-06-06-windowed-recency-strategy-replay-design.md`.
- **Hitters** (`BlendedSource`): `seasonWeight = approxPA / (approxPA + 250)` where `approxPA = GP × 3.8` and **GP is games in the trailing 30-day window** (caps ~26), so recent weight tops out near ~28% but reflects only recent form; base weight = `max(1 - seasonWeight, 0.30)`, never below 30%. `PtsPerGameSource` interface (type assertion) lets the optimizer use pre-computed values.
- **Pitchers** (`PitcherBlendedSource`): role-aware stabilization — SP equal weight at 15 GP, RP at 25 GP; base floor 35%. Requires minimum GP before blending. GP here is season-to-date (YTD snapshot). `PitcherPtsPerGameSource` interface.

**`internal/prospects`** — monitors minor league prospects across MLB transactions, MiLB performance breakouts, and prospect ranking sources (MLB Pipeline primary, FanGraphs fallback). Produces a daily prospect report in the GHA job summary with call-up alerts, hot streak detection, free agent watch, and upgrade recommendations. Separate from roster alerts (which detect slot mismatches); this focuses on external data to find new players to pick up. Rankings are cached in `.cache/` (168h default TTL). Breakout detection uses level-adjusted thresholds (AAA/AA/A-ball). Transaction tracking uses a cursor to avoid duplicate alerts across runs.

**`internal/gscheck`** — league-wide GS violation checker. `RunGSCheck` fetches all scoring periods and teams via `getStandings`, fetches the real GS min/max for the completed period via `GetGSLimits` (no static fallback — a live-fetch error sends a Pushover alert and aborts the run; no configured max means a clean no-op), iterates every team to tally active-slot pitcher GS, detects violations (GS > max or GS < min), and sends a Pushover notification. The `gs-check` CLI command requires `GS_TRACKING_ENABLED=true`, `PUSHOVER_GROUP_KEY`, and `PUSHOVER_API_TOKEN` to be set before running (a clean no-op message if tracking is disabled, not an error).

**`internal/transactions`** — trade monitor. `CheckTrades` fetches recent Fantrax league trades (last 24 hours) via `GetRecentTrades`, groups them by `TradeGroupID`, values each side using HKB player rankings, and sends a Pushover notification with the trade report. Uses normalized name matching (lowercase, stripped suffixes) to join Fantrax player names to HKB data. Requires `PUSHOVER_USER_KEY` and `PUSHOVER_API_TOKEN` for notifications (skips if not set).

**`internal/statcast`** — leaf owner of Baseball Savant data and the derived buy-low / hot **Statcast Signals** (imports only `internal/cache` and `internal/archive`). Holds the **Statcast Bundle** (`Bundle` — the five Savant expected-stats / quality-of-contact slices joined by MLBAM ID), `LoadBundle` (cached at `CacheTTL` = 24 h; cache-key prefix `keySavant` = `"savant"`, unchanged from its former home so existing `.cache/savant-*` entries still hit), the signal engine (`Signal`, `TagHitter`/`TagPitcher` returning `(Signal, SignalMetrics)`), `Thresholds`/`DefaultThresholds`, and `ArchiveArtifacts` (raw-CSV archival). Extracted from `internal/waivers` so `waivers`, `claims`, and `recap` depend on this data leaf — not on each other's command package. `TagHitter`/`TagPitcher` return a `SignalMetrics` of the facts behind the classification; callers own identity/report fields (e.g. `waivers.Candidate` embeds `SignalMetrics`).

**`internal/waivers`** — Statcast-driven waiver wire audit. `Run(ft, today, opts)` fetches the full Fantrax player pool, filters to MLB free agents (FantasyStatus `"FA"` / `"W*"` / empty, excluding `MinorsEligible`), resolves names → MLBAM IDs via `playername.ResolveMLBAMIDs`, and joins each FA against (a) FanGraphs projections (configured system; default `depthcharts`) from `projections.LoadBattingProjections`/`LoadPitcherProjections` for the per-game point projection, and (b) a `statcast.Bundle` of five Baseball Savant CSVs for signal tagging (the Savant loader and signal engine live in **`internal/statcast`**, not here — see below). Two signals fire per player: **BUY-LOW** when xStats outpace surface stats (hitter: `xwOBA - wOBA ≥ 0.030` AND barrel ≥ 9 AND hard-hit ≥ 42; pitcher: `era - xera ≥ 1.00` AND `xwoba ≤ 0.310`) and **HOT** when recent production is backed by quality (hitter: 14d `wOBA ≥ .380` AND `xwOBA ≥ .360` AND season barrel ≥ 8; pitcher: 30d `era ≤ 3.20` AND `xera ≤ 3.50`). When both fire, the row is tagged `BOTH`. Sample guards: hitter season ≥80 PA, hitter 14d ≥20 PA, pitcher season ≥100 TBF, pitcher 30d ≥50 TBF. Defaults live in `statcast.DefaultThresholds()`; tests override. The design separates concerns deliberately — Statcast picks WHO surfaces, FanGraphs scores HOW MUCH — so the league's own `ScoringWeights` (via `ExpectedPtsFromProj`) drive the ranking and FAs without a FanGraphs projection are skipped (accepted v1 limitation). Output: stdout (always), GHA markdown summary (when `GITHUB_STEP_SUMMARY` set), Pushover (when `!DryRun` and creds set; `formatPushover` truncates to fit the 1024-char limit). CSV fetchers (in `statcast/load.go`) have URLs as `var` for test override; column lookup is by lowercased name with aliases so Savant header drift doesn't break parsing. CSVs are cached at 24h TTL (`statcast.CacheTTL`) via `cache.FileCache[T]` keyed by year + window-end-date.

**`internal/claims`** — league-wide CLAIM/DROP recap. `Run(ft, today, opts)` fetches all recent Fantrax transactions via `GetAllTransactions` (cached as `fantrax-all-transactions-<leagueID>`, `todayTTL`), filters to CLAIM and DROP events since the last run using a cursor persisted at `.cache/last-claims.json`, and pairs each CLAIM+DROP by `Transaction.ID` to reconstruct complete moves. Each picked-up player is valued via HKB rankings (reusing the same normalized-name join as `internal/transactions`) to produce a "value gained" metric and a per-team daily leaderboard. When `!opts.NoSignals`, picked-up players are enriched with Statcast signal tags (BUY-LOW / HOT / BOTH) by joining against the `statcast.Bundle` and FanGraphs projections — the same enrichment path as `internal/waivers`. Notable drops (HKB value above `opts.DropsMin` threshold) are flagged with a watch note. Results are written to an audit ledger at `.waivers/claims/<date>.json` (directory created on first write). On an empty claim window the command is a no-op. Sends Pushover when `!opts.DryRun` and creds are set. `opts.Since` overrides the cursor for one-off historical recaps without resetting the cursor state.

**`internal/backtest`** — grades past work against actual outcomes. Two analyses:
- **Lineup grading**: for each past day, computes an actual-points total (sum of FPts for active-slot players) and a hindsight-optimal total (the existing optimizer run against a `hindsightSource` that returns each player's actual FPts as pts-per-game via the `PtsPerGameSource` / `PitcherPtsPerGameSource` interfaces). `Gap = actual - optimal`; negative means points left on bench. SP-eligible pitchers who actually appeared are fed to the optimizer as "probable starters" so the 0.10x non-starter discount doesn't apply in hindsight.
- **Projection grading**: checks `.backtest/snapshots/<YYYY-MM-DD>.json` for archived per-player projection values written by `optimize`. Each snapshot row (`backtest.SnapshotPlayer`) records the projected pts/game plus look-back fields — `was_started`, `slot`, `locked`, `role`, and `eligibility` (position IDs) — so error can be sliced by position and lineup decision. When present, grading compares against actual FPts for an overall MAE/Bias/RMSE report, a per-position MAE table (buckets C/INF/OF/UT for hitters via eligibility precedence C>INF>OF>UT, and SP/RP for pitchers via role), and the top-10 signed-error misses (ordered most-over-projected first so ramp-up patterns surface). When a day's snapshot is absent, the day is marked `source="missing"`. When a snapshot exists but its `generated_at` falls on a different ET calendar day than the date it projects — a `--matchup` pre-write that was never refreshed on the day itself (e.g. every hourly run that day failed) — the day is marked `source="stale"` and excluded from the rollup, since grading a multi-day-old forecast inflates apparent projection error (`sameETDate` guard in `backtest.go`, keyed on ET since that's the zone the daily runs and matchup-day boundaries are anchored to; a zero `generated_at` predates the field and is graded as before). `FormatReport` prints an "Excluded from grading: N stale, M missing" line listing the dates, so a window thinned by stale/missing days is visible rather than silently shrinking the sample. The hourly `lineup.yml` workflow runs `optimize` (which writes snapshots by default on non-dry-run) and persists `.backtest/snapshots/` in the GHA cache (alongside `.cache`), so production accumulates a snapshot per day from the most recent hourly run. Local backtest runs that need the snapshots have to either (a) replay the workflow's cache or (b) run `optimize --archive-projections` themselves.

Per-day FPts come from `fantrax.DailyFantasyPoints` (in `internal/fantrax/daily_fpts.go`), which walks a period range and diffs consecutive YTD snapshots via `playerStatsFromTables`. This covers both `scGroup=10` (hitting) and `scGroup=20` (pitching). The Fantrax roster API is requested with `StatsType=1` (MLB stats — real per-player season totals) rather than `StatsType=2` (Fantasy Team — team-credited only) so reserve players' production is visible to hindsight callers; without this, the optimal lineup calc in the backtest/recap can never recommend a benched player and efficiency % stays artificially compressed (~95–99% across the league).

Players seen for the first time in the window have their `(deltaFP, deltaGP)` zeroed so the cumulative pre-window YTD doesn't leak in as same-day production (a waiver pickup with a 200-FPts season-to-date YTD would otherwise inject 200 phantom points on the day they appear). The zeroed row is flagged via `DayPlayerFP.NeedsBackfill`; `Client.BackfillDailyFPts` (in `internal/fantrax/mlb_backfill.go`) resolves the flag by fetching the player's MLB statsapi game log for that date and computing FPts from raw stats × the league's `ScoringWeights` — bounded to a handful of API calls per recap week, soft-fails to "leave zero" on any error. Two-way crossings (Ohtani moving between the hitters and pitchers tables) are handled the same way: `diffYTD` detects the `prevOther` fallback (role-specific YTDs can't be meaningfully subtracted), zeroes the diff, sets `NeedsBackfill`, and the backfill computes the role-correct same-day FPts. Both fixes verified against the Fantrax-authoritative `Matchup.HomeTeam.Points` for week 8 (recap matched all 10 team totals exactly). Per-period snapshots are cached at `.cache/fantrax-roster-stats-<teamID>-<period>.json` with a 30-day TTL since past periods are immutable. A parallel cache `.cache/fantrax-pitcher-gs-<teamID>-<period>.json` (same TTL) backs `GetTeamPitcherStarts`, used by the recap to enumerate every active-slot SP start; it's optional (pass `cacheDir=""` / `cacheTTL=0` to bypass, which the live `gs-check` path does because it always wants fresh data), and when the cache hits, the per-day 200ms upstream-throttle in `GetTeamPitcherStarts` is skipped.

Snapshot writing is default-on for real (non-dry-run) `optimize` runs so the cron accumulates backtest data automatically. Dry-run stays side-effect-free unless you opt in with `--snapshot` (the older `--archive-projections` flag and `BACKTEST_ARCHIVE=1` env var are kept as backward-compatible aliases). Snapshots are rewritten if the same date is optimized twice — last write wins, which is fine for the hourly lineup workflow: the final write of the day captures the most recent projection state for that date, and is the version `backtest` grades against. If that final same-day write never lands (the day's runs all failed), the snapshot is left as an earlier `--matchup` pre-write and the stale-snapshot guard above excludes it from grading.

**Multi-system capture (`shadow`)** — `cmd/shadow.go` drives the full `internal/lineuprun.Run` pipeline (matchup adjustment included) once per RoS system in dry-run, capturing every system's projection side-by-side for the model-comparison report — no lineup applied, no notification. It shares the orchestration engine with `optimize` (zero pipeline duplication) by calling the same `Run` entry point in a loop: each iteration calls `initApp` fresh and builds an `Options` with `ProjectionSystem: sys`, `SnapshotRoot: systemSnapshotDir(shadowSnapshotRoot, sys)` (instead of the flat `.backtest/snapshots/`), and `ForceSnapshot: true`, looping over `shadowSystems` (`steamer-ros`, `depthcharts-ros`, `thebatx-ros`, `atc-ros`). `grade` then loops the same `shadowSystems`, calling `RunProjectionAnalysis(days, systemSnapshotDir(...))` per system (actuals fetched once) and writing each system's rows under its `system=` partition. The hourly `optimize`'s flat snapshot is untouched (it still feeds the local `backtest` command). The EventBridge `Shadow` schedule runs **23:40 UTC** — late ET evening but still the **same ET calendar day as the captured date**, so the snapshot's `generated_at` passes the `sameETDate` stale-guard and is graded by the next 13:30 UTC `grade` run. `.backtest/snapshots-systems/` rides the existing recursive `.backtest/` S3 sync (`del=false`), so the shadow task's `sync-up` and grade task's `sync-down` hand it across Fargate tasks with no entrypoint change.

**`internal/recap`** — Sleeper-style weekly recap. `recap.Run(ft, opts)` aggregates all 12 (or however many) teams in parallel via `errgroup`: for each team it pulls `DailyFantasyPoints` for the matchup week and runs `backtest.RunLineupAnalysis` to compute actual + hindsight-optimal totals, plus `GetTeamPitcherStarts` (a sibling to `GetTeamGS` in `internal/fantrax/pitcher_starts.go`) to enumerate every active-slot SP start with its FPts. H2H pairings come from `GetAllMatchupEntries` (a passthrough wrapper added on `*fantrax.Client`); team weekly scores are aggregated from daily FPts (deterministic, doesn't depend on parsing the upstream `MatchTeam.Total`). Award functions in `awards.go` are pure and unit-tested. **League Leaders** (`leaders.go`, `buildLeaders`) ranks every rostered player league-wide (from `GetFullPlayerPool`, filtered to `FantasyTeamID != ""`) by season-to-date wOBA (hitters) and FIP (pitchers): names → MLBAM IDs via `playername.ResolveMLBAMIDs`, wOBA joined from `statcast.LoadBundle`'s qualified-hitter expected-stats CSV, FIP computed from a batched MLB statsapi season-pitching fetch (`mlbSeasonPitchingURL`, `parseIP` handles `.1`/`.2` thirds) with the FIP constant solved from the rostered-pitcher pool so displayed values land near league average (ranking is constant-invariant). Pitchers need ≥`fipMinIP` (30) innings. The whole path soft-fails to nil — the section is `{{if}}`-guarded out when empty, so archived/credential-less renders stay clean. The single-day leaderboards and league-leaders board both badge each row with the owning team's logo (`PlayerLine.OwnerTeamID` / `LeaderLine.OwnerTeamID` → `teamLogo`) instead of a rank number (rank implied by order); default board size is top 5 (`--top`). The season leaderboard (`RenderSite`) follows the same rank-implied-by-order convention: award categories drop the numbered chip (owner logo only), and the season Shellings list badges each row with the pitcher's **MLB-club** logo via `mlbTeamLogo` (abbrev → statsapi team ID → mlbstatic "spots" PNG endpoint; `mlbTeamAbbrevToID` covers both canonical and upstream-variant abbreviations, falling back to a text chip on an unknown code). Two render helpers centralize the avatar markup so award cards stay terse: `teamBadge(logos, id, name)` emits the fantasy-team logo-or-initial chip (used left of the name in Most/Least Efficient and every League Award card — winner-side cards pass `WinnerID`), and `mlbBadge(abbrev)` emits the MLB-club logo-or-text chip (used left of the pitcher name in the Pitching Highlights best/worst cards). Both return trusted `template.HTML` (our own URLs; names HTML-escaped). The renderer (`render.go` + embedded `template.html`) emits a single self-contained HTML file via `Render` (no nav) or `RenderSite` (with a cross-week dropdown). `recap.RunSite(ft, sopts)` (`internal/recap/site.go`) drives the multi-week build: it enumerates every completed matchup week via `GetMatchupWeekByNumber`, calls `Run` for each, and writes `dist/week-NN.html` plus `dist/index.html` (mirror of the latest week). Each page carries a `<select>` dropdown of all weeks. The `recap-site` CLI command exposes this for the GitHub Pages workflow.

**Week-completion signal** — `completedMatchupWeeks` and the `recap` command's range resolver treat a week as renderable when `weekEnd < today` (strict, date-only) **OR** `weekEnd == today AND every MLB game on that final day is over**. The same-day path lets a Sunday-ending week render the same evening once its games conclude. The completion check is `schedule.Client.AllGamesFinalOn(date)` (MLB statsapi `abstractGameState == "Final"`, plus Postponed/Cancelled via `detailedState`; vacuously true when no games are scheduled that day). The Fantrax weekly `Matchup.*.Points` field is **not** used for this — it's a **running in-week score** (non-zero from the week's first games onward), so an earlier `MatchupWeekIsFinal`/`IsMatchupWeekFinal` heuristic that keyed off `Points > 0` was removed for falsely flagging in-progress weeks (it rendered week 10 on its live final Sunday). `completedMatchupWeeks` takes a `dayCompletionChecker` interface (satisfied by `*schedule.Client`) so the cutoff logic is unit-testable without network; on a schedule lookup error the same-day week is conservatively excluded.

**WP model** — `wp.go` exposes `ComputeWPCurve`, `LeadChangeCount`, `MinWinnerWP`. `ComputeWPCurve` is a Go port of the formula Fantrax runs client-side in its live-scoring Angular bundle (`models.CalculateWinProbability` in go-fantrax): deterministic, no Monte Carlo, no sigma. At each of 8 points (pre-week + 7 day-ends) it feeds cumulative actual FPts, a **live-adjusted** weekly projection (`actual_so_far + remaining_days × HomeMeanDaily`, mirroring how Fantrax recomputes `calculatedProjectedTotalsMap` intra-week — without the live adjustment the projection ratio stays flat all week and the chart degenerates to 50% then a snap to 100%), and a uniform `timeLeft = 7 - i` for both teams into the Fantrax formula. Pre-week WP[0] reflects projection ratio (a 60/40 projected favorite starts at ~84/16, not 50/50). Final WP[7] is always 100/0 or 0/100 — on an exact-tie final the Fantrax formula awards the win to "away" (faithful port; ties never happen in practice with float FPts).

**Game of the Week** — always featured at the top of every week (no skip for quiet weeks). Picked by `GameOfTheWeek(curves, matchups)` using `score = LeadChanges + (1 - minWinnerWP)` — the "weeeeeh factor": back-and-forth swings plus how deep the eventual winner sank mid-week. Ties broken by smallest final margin then home `TeamID` asc. Ties (no winner) score with the comeback term zeroed. The hero chart is rendered as a 380×140 SVG with mirrored 100/75/50/75/100% y-axis labels, half tints (green=home favored on top, red=away favored on bottom), team name labels in their respective halves, and dated x-axis ticks. The matchup-results list marks the chosen game with a ⭐ badge.

**Comeback award** — winner whose mid-week WP dropped below `comebackThreshold = 0.30` (eventual-winner's minimum during `Points[1..6]`). Feeds into `AggregateSeasonAwards`. Game of the Week is intentionally excluded from the season leaderboard — it shows at the top of each week's page, so cumulative counts would be redundant.

**`internal/analysis`** — the durable **Analysis Store** (model-performance history, opposite lifecycle to the ephemeral Cache). The daily `grade` command reuses `backtest.RunProjectionAnalysis` to materialize **Graded Snapshots** (`analysis.GradeRow`: per (date, player) projected/actual/diff + bucket/role/is_pitcher) and writes them as NDJSON to `analysis/grades/dt=YYYY-MM-DD/system=<sys>/grades.ndjson`. **System is a Hive partition owned by the object path, not the NDJSON body** (`GradeRow.System` is `json:"-"`) so it never collides with the Athena `system` partition column; writers take it as a `WriteGrades(date, system, rows)` argument and readers populate it from the key via `SystemFromKey`. Legacy partitions written before the system dimension (`grades/dt=X/grades.ndjson`, no `system=` segment) are read back under `analysis.LegacySystem` (`"depthcharts-ros"`, the system the bot ran) so the detail dashboard keeps its pre-migration history. `analysis` defines `GradeRow`/`MarshalNDJSON`/`NewFileWriter`/`Writer`/`SystemFromKey`/`LegacySystem`; the S3 adapter is `internal/analysis/s3grades` (aws-sdk-go-v2, isolated like `cachestore/s3store`). `cmd/grade.go` writes via S3 when `STATE_BUCKET` is set, else local `.analysis/`, looping the 4 shadow systems (actuals fetched once, graded per system). The read side mirrors the write side: `analysis.Reader` + `FileReader` glob both `dt=*/system=*/grades.ndjson` and legacy `dt=*/grades.ndjson` from `.analysis/`; the S3 adapter `s3grades.Reader` lists+streams the `analysis/grades/` prefix the same way. `cmd/projection-site.go` wires the appropriate reader by `STATE_BUCKET`. The CDK-managed Athena table `rosterbot_analysis.grades` (partition projection on `dt` + `system`-enum, no crawler) makes the store queryable for model auditing (MAE/bias by position **and by system** over time). Snapshots persist because `entrypoint.sh` syncs `.backtest/` (under the `backtest/` prefix). See `docs/adr/0001-s3-not-db-for-cache.md` and `docs/aws-deployment.md`.

**`internal/archive`** — stdlib-only leaf package for durable daily snapshots of ephemeral upstream data that can't be re-fetched later. Defines `Artifact{Filename, Bytes}`, a `Source` interface (`Fetch(ctx, date) ([]Artifact, error)`), `FuncSource` (adapter), `Writer` (writes raw bytes to `.archive/<source>/dt=YYYY-MM-DD/<filename>`), and a shared `Get` raw-fetch helper. Concrete source implementations live in each provider package: `ArchiveArtifacts(ctx, date)` in `internal/{hkb,projections,statcast,prospects}` covers HKB rankings, all four FanGraphs RoS systems (steamer-ros/depthcharts-ros/thebatx-ros/atc-ros, batting + pitching = 8 blobs), the five Baseball Savant CSVs (season plus 14d/30d rolling windows), and the FanGraphs prospect board. The `archive` CLI command (`cmd/archive.go`) runs all sources with per-source isolation — one failure doesn't block others; exits non-zero only if every source fails. Supports `--date YYYY-MM-DD` (default today UTC) and `--dry-run` (fetch + print sizes, write nothing). `.archive/` syncs to S3 `archive/` prefix via `statePairs` in `cmd/sync.go`, uploaded append-only (`del=false`, keep-forever). Distinct from the TTL cache (which overwrites on expiry) and additive to the projection snapshot/Analysis Store.

**`internal/report`** — pure aggregation package (no I/O, no rendering). Consumes `[]analysis.GradeRow` and produces a `Model`. The **detail dashboard** (`Views`/`Trends`, keyed `"window|role"`) is computed from the **`detailSystem` slice only** (`= analysis.LegacySystem` = `"depthcharts-ros"`, the bot's production system) so its meaning is unchanged; each `View` holds a `Scorecard` (MAE/Bias/RMSE/N), `[]PositionRow`, `[]CalibPoint`, `[]Miss`, plus a `[]TrendPoint` and rule-based `[]Insight`s. On top of that sits the **system-comparison panel**: `Compare` (keyed `"window|role"` → `[]SystemScore` ranked by MAE ascending, lowest-MAE-with-data flagged `Best`) and `CompareTrends` (keyed `"window|role"` → system → `[]TrendPoint`) span **every captured system**, driving the headline ranking table + overlaid per-system MAE trend lines. `Aggregate` normalizes empty-`System` rows to `detailSystem` (matching the reader's legacy attribution) so un-attributed input still feeds the detail views. `rankSystems`/`filterSystem`/`distinctSystems` are the comparison helpers. There is no `Render`/`template.html` anymore (Dashboard v2 deleted both) — the `Model` is serialized straight to JSON and the dashboard SPA owns presentation. The daily `ProjectionSite` EventBridge schedule (`cron(0 15 * * ? *)`, ~90 min after `grade`) runs `projection-site`, which reads from the Analysis Store, calls `report.Aggregate`, and writes `./report/model.json`, which `entrypoint.sh` syncs under `DASHBOARD_BUCKET`'s `report/` key prefix — the same bucket/`DashboardCdn` CloudFront distribution as the private dashboard SPA (`web/dashboard/`). The SPA's `projections.js` fetches `model.json` and renders the scorecard/trend/calibration/misses/comparison views client-side (vendored Chart.js, ported from the deleted template's inline script — no recomputation, just reshaping the precomputed `Model` into DOM), surfaced as the native "Projections" tab (no iframe). Distinct from the recap's `SITE_BUCKET`, which remains its own separate site.

**`internal/teamvalue`** — the durable **Team Value Store**: each fantasy team's aggregate HKB dynasty value for one day, as date-partitioned NDJSON (`dt=YYYY-MM-DD/values.ndjson`). Mirrors `internal/analysis` structurally but is single-entity (no `system` dimension), so the partition key is just `dt=` and the entity name lives in the S3 prefix (`analysis/team-values/`) / local dir (`.teamvalue/`). `teamvalue.Row` stores value as **four leaf cells** — hitter/pitcher × MLB/minors — plus matching count leaves and `RosteredCount`/`MatchedCount` (join-coverage transparency); `TotalValue`/`MLBValue`/`MinorsValue`/`HitterValue`/`PitcherValue` are derived methods. `teamvalue.Aggregate(date, pool, hkbPlayers, teamNames, teamLogos)` joins each rostered player (`FantasyTeamID != ""`) to its HKB `Value` by `playername.Normalize` (the same join as `claims`/`transactions`), buckets by Fantrax `MinorsEligible` (minors) and `positions.IsPitcherSlot` (pitcher; a two-way player with any pitcher eligibility resolves to pitcher), and denormalizes team name + logo into each row so the read+render path needs no Fantrax call. Unmatched rostered players increment `RosteredCount` only (value/count leaves are matched-only). `Writer`/`FileWriter`/`MarshalNDJSON`/`ObjectKey` + `Reader`/`FileReader`; the S3 adapter is `internal/teamvalue/s3teamvalue` (aws-sdk-go-v2, isolated like `analysis/s3grades`). The series **accumulates forward** — HKB has no history and rosters aren't archived, so past compositions are unrecoverable; there is no backfill (see `docs/adr/0002-team-value-store-accumulates-forward.md`).

**`internal/valuereport`** — pure aggregation (mirrors `internal/report`; no I/O, no rendering). `BuildModel([]teamvalue.Row)` produces a `Model` (per-team `Series` of the four leaves + a `Latest` total-descending standings snapshot; stable per-team colors by sorted TeamID). There is no `Render`/`template.html` anymore (Dashboard v2 deleted both) — the `Model` is serialized straight to JSON. The `team-values` command (`cmd/team-values.go`) is the daily producer — fetches HKB + `GetFullPlayerPool` + `GetScoringPeriodsAndTeams`, calls `Aggregate`, and writes today's partition (S3 when `STATE_BUCKET` set, else `.teamvalue/`); it runs on the `TeamValues` EventBridge schedule (`cron(30 14 * * ? *)`, after `Archive`'s HKB refresh, before `ProjectionSite`). `cmd/projection-site.go`'s `renderValueSite` reads the store (reader chosen by `STATE_BUCKET`) and writes `value.json` alongside `model.json` — both published under `DASHBOARD_BUCKET`'s `report/` prefix — soft-failing independently so a team-value hiccup never blocks the accuracy dashboard deploy. The SPA's `value.js` fetches `value.json` and renders the multi-team Chart.js time plot with a **metric selector** deriving Total/MLB/Minors/Hitter/Pitcher client-side, a team legend toggle (+ All/None), and a current-standings table with logos and join coverage, surfaced as the dashboard SPA's native "Value" tab (no iframe).

**`internal/notify`** — notification helpers. `SendPushover` sends push notifications via the Pushover API. Self-contained function taking explicit parameters (no config dependency).

**`internal/roster`** — `CheckRoster` scans the full roster for slot mismatches (healthy players in IL, called-up players in Minors, injured/minor-leaguers in active slots). Suppresses alerts when IL/Minors slots are full. Separate from prospect report — this is about current roster hygiene.

**`internal/schedule`** — hits `statsapi.mlb.com` for game schedule and probable pitchers. `TeamsPlayingOn` returns a `map[string]bool` of playing team abbreviations. `ProbableStarters` returns normalized pitcher name → team abbreviation. Both URLs are `var` (not `const`) to allow test overriding.

**`internal/optimizer`** — pure functions, no I/O. Two parallel optimizers:
- **Hitters** (`OptimizeLineup`): backtracking with pruning to find globally optimal slot assignment maximizing total expected points. Checks `PtsPerGameSource` (type assertion) before falling back to `expectedPts`. `EligibleForSlot` in `fantrax/client.go` handles UT (accepts any hitter) and INF (accepts 1B, 2B, 3B, SS — not C).
- **Pitchers** (`OptimizePitcherLineup`): sorts by hasGame → expectedPts → ID, then assigns to slots. Uses probable starter data to determine if SPs start; when no probable data is available (future dates), SPs default to "has game" if their team plays. Accepts an optional `*GSBudget` for weekly game-start limit awareness.

**Scoring model** — this league scores: `1B`, `2B`, `3B`, `HR`, `RBI`, `R`, `BB`, `SB`, `CS`, `HBP`, `SO`, `GIDP`, `XBH`, `TB`, `CYC`. The stat→points algebra lives in **`internal/scoring`** (the single source): `ApplyHitter`/`ApplyPitcher` take a neutral `HitterLine`/`PitcherLine` of raw counts, derive `1B = H - 2B - 3B - HR` (floored at 0), `XBH = 2B + 3B + HR`, `TB = 1B + 2×2B + 3×3B + 4×HR`, and apply the league weights. The package imports nothing else (so both `fantrax` and `projections` can depend on it without a cycle); `fantrax.ScoringWeights` is a type alias for `scoring.Weights`. Callers adapt their source to a stat line: `projections.ExpectedPtsFromProj`/`PitcherExpectedPtsFromProj` build a line from a projection and divide by games (scoring is linear, so per-game = total/G); the `mlb_backfill` game-log scorers build a line and don't divide (single game). `optimizer` and `waivers` consume these rather than re-implementing the math.

**GS budget** — weekly game-start limit awareness (`GS_TRACKING_ENABLED` env var, false = disabled). When enabled, the pitcher optimizer resolves today's scoring period and fetches its real GS max via `GetGSLimits`, gating SP starts to avoid exhausting the weekly GS allocation on low-value starters while better aces pitch later in the matchup week. A live-fetch error sends a Pushover alert (`PUSHOVER_USER_KEY`) and disables the gate for that run rather than falling back to a stale static number; no configured max just disables the gate silently (nothing to track).
- **Matchup week boundaries** derived from `GetAllMatchups()`: consecutive daily scoring periods where the team faces the same opponent form a matchup week. Computed in `fantrax/matchup_weeks.go` via `MatchupWeekBounds`.
- **Past GS counting**: for each past day in the current matchup week, the `ProbableStarters` API is checked to count how many rostered SPs started.
- **Future demand forecasting** uses a hybrid approach: days with confirmed probable starters use exact counts; days without probables estimate `roster SPs whose team plays / 5` (standard 5-man rotation).
- **Value-ranked gate** (`optimizer/gs_budget.go`, `applyGSGate`): when remaining budget can't cover every planned start (today + future confirmed + future estimated), pools all of them into one ranking by expected points and keeps only the top `remaining` by value — it does not split the budget proportionally by day/count. Future days without announced probables contribute `ceil(Estimated)` placeholder entries valued at the roster-wide mean SP projection (a real projection, not a scrub), so a fractional rotation-math estimate (e.g. 0.2 expected starts) still competes as a full-value placeholder against today's confirmed starters. Ties prefer today's starters (a certain start) over future ones, then break on player ID. Starters that don't make the cut have `IsStarter` flipped to false, applying the existing 0.10x non-starter discount. Uses `eps = 1e-9` for float comparison consistency. Locked players are exempt from suppression (their GS is already decided one way or the other).
- The gate only applies to today's optimization (the daily GHA run). Future dates in `--dates` ranges are optimized without the gate since each day gets its own run.
- The `--matchup` flag on the optimize command resolves to all remaining days in the current matchup period (from today through the matchup week end).

**`internal/lineuprun`** — the shared orchestration engine behind both the `optimize` and `shadow` CLI commands: fetches rosters/projections, runs the hitter and pitcher optimizers per date (the parallel `errgroup` pass over `cfg.Dates`), archives projection snapshots, publishes the read-only lineup API JSON, and prints + applies the resulting moves. `Run(ft, cfg, opts)` takes an explicit `Options` struct (projection system, `--dates`/`--matchup` resolution flags, pipeline-detail toggle, snapshot behavior, cache/verbosity) — there is no package-level mutable state shared between callers. `cmd/optimize.go` (a single real/dry-run pass) and `cmd/shadow.go` (one dry-run capture per RoS system, looped) are both thin cobra adapters that parse flags, call `initApp`, build an `Options` value, and call `Run`. `Options.SnapshotRoot` is the exact directory a run's projection snapshots are written into — always set explicitly by the caller (`optimize` passes the flat `.backtest/snapshots/` path; `shadow` passes a per-system partition each loop iteration) — and `Options.ForceSnapshot` is a separate explicit boolean (`shadow` sets it so a capture run always writes regardless of dry-run). This replaced a former package-level `captureSystemRoot` var in `cmd/optimize.go` that `cmd/shadow.go` mutated to redirect snapshot output (its emptiness used to double as both "not in shadow mode" and "use the default directory"), plus a parallel `lastProjectionLoadResult` var that let `shadow` observe per-system projection-data availability only because `runOptimize`'s signature was fixed by cobra — `Run` returns a `Result{HittersNoData, PitchersNoData}` instead (**rosterbot-zbi**).

**`internal/progress`** — live run-progress recording, layered on top of the existing terminal progress bar (`Progress.Start`/`Done`/`Warn`). `Recorder` is a nil-safe package-level hook (mirrors `lineupapi`'s `OutputRecorder` pattern): every phase transition also builds a `Snapshot` (`{phase, pct, phases:[{name,state}], status:"running", updated_at}`, ordered per `PipelinePhases`/`phaseWeight`) and hands it to `Recorder` if one is installed. `cmd/progress.go`'s `installProgressRecorder()` wires it from `RUN_ID`, writing to `runs/<id>/progress.json` — S3 via `internal/lineupapi/s3lineup.NewProgress` (`STATE_BUCKET` set) or locally via `lineupapi.NewFileProgressStore(".lineup/progress")` — and is a best-effort no-op if `RUN_ID` is unset or the store errors. `internal/lineupapi`'s `GET /v1/runs/{id}/progress` route (`ProgressStore.GetProgress`) serves that snapshot, `404` when absent. **Run status stays ledger-owned** — `progress.json` only adds phase detail on top of the `runledger/`-backed `GET /v1/runs`/`GET /v1/runs/{id}` status, never replaces it. Only `optimize` calls into `internal/progress` today (it's wired through `internal/lineuprun`'s pipeline phases); the other 8 allowlisted jobs emit no progress file, so the dashboard SPA's live hero (`web/dashboard/live.js`) falls back to an indeterminate bar for them on the `404`. No new IAM: writes ride the existing `runs/` S3 prefix the run ledger and per-run output already use.

## Idempotency

The optimizer must produce identical output given the same inputs. Key invariants:
- **Stable sort**: player ranking uses player ID as tiebreaker (`scored[i].Player.ID < scored[j].Player.ID`) so equal-scoring players always appear in the same order.
- **Epsilon comparison**: the backtracking optimizer uses `eps = 1e-9` for floating-point comparison to avoid flip-flopping between equivalent assignments.
- **Minimal changes**: when two assignments produce the same total points (within epsilon), the optimizer prefers the one with fewer roster moves (`changes < bestChanges`).
- **Period-specific roster**: for future dates, the optimizer fetches the roster for that period (`GetHitterRosterForPeriod`) so it sees already-applied lineups. A second run with the same inputs produces "No changes needed".
- **Verification**: after any optimizer change, run the command twice with the same inputs and confirm the second run shows "No changes needed" for all dates.
